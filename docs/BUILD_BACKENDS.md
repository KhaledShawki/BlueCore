# Build Backends

BlueCore uses Premake as the single source of truth for project structure. All generated build files and binaries are written under the `out/` directory.

## Backend Policy

| Environment          | Primary Backend          | Notes |
|----------------------|--------------------------|-------|
| Windows development  | Visual Studio (`vs2026` / `vs2022`) | Full solution with helper targets and static/shared matrix support |
| macOS development    | Ninja                    | Fast local builds. Static `x64` variant by default |
| Linux development    | Ninja                    | Fast local builds and CI. Static `x64` variant by default |
| Unix fallback        | `gmake2`                 | Use when diagnosing Ninja generator or toolchain issues |
| Tooling / IDEs       | `compilecommands`        | Used for clangd, CLion, and VS Code code intelligence |

## Ninja Policy

Premake’s Ninja backend currently generates only the static `x64` build platform. It does not support BlueCore’s per-platform static/shared switching in a single project file.

Use Visual Studio or `gmake2` when you need the full `x64` + `x64_DLL` matrix until the Ninja model is extended.

## Commands

**macOS**

```bash
./scripts/run-tests-macos.sh Debug_x64
```

**Linux**

```bash
./scripts/run-tests-linux.sh Debug_x64
```

**Manual Generation (macOS example)**

```bash
./scripts/premake-macos.sh ninja \
    --toolchain=clang \
    --blue-platforms=macos \
    --blue-build-platforms=x64 \
    --memory-backend=system \
    --blue-startup=BlueRunTests

ninja -C out/build/ninja BlueRunTests_Debug_x64
```

## Output Directories

All generated artifacts are written under `out/`:

```text
out/build/<backend>                    Generated project files
out/bin/<system>/<platform>/<config>   Executables and libraries
out/obj/<system>/<platform>/<config>   Object files
```

Nothing generated by the build system should be written into `modules/`, `apps/`, `tests/`, or `build/`.

## macOS SDK Handling

On macOS, the Ninja flow uses Apple Clang through `xcrun` and injects the active macOS SDK into compile and link commands. This ensures that headers such as `pthread.h`, `errno.h`, and libc++ headers are visible, even when using non-system Clang installations.

The test scripts create temporary compiler wrappers under `out/tools/macos-clang/` and prepend this directory to `PATH` before invoking Ninja. These wrappers are generated artifacts and are safe to delete along with the rest of `out/`.
